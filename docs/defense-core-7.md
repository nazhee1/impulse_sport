# 7 опорных блоков для защиты

Этот файл нужен, чтобы не учить десятки вопросов по отдельности. Если вы запомните 7 блоков ниже, то сможете ответить почти на все стандартные вопросы комиссии.

## Как отвечать на любой вопрос

Используйте простую схему из 3 шагов:

1. Что это такое.
2. Зачем это нужно в моем проекте.
3. Как это реализовано или предусмотрено у меня.

Пример:

REST API — это способ обмена данными между клиентом и сервером. В моем проекте он нужен для передачи данных о товарах, корзине и заказах. В текущей версии он спроектирован как основа для backend-части.

## Блок 1. О чем проект

Мой проект — это информационная система интернет-магазина спортивных товаров. Она предназначена для просмотра каталога, поиска и фильтрации товаров, добавления их в корзину и оформления заказа.

Это ваш главный ответ на вопросы:

- какая тема;
- что вы делали;
- что это за система;
- в чем суть проекта.

## Блок 2. Зачем проект нужен

Проект нужен для автоматизации онлайн-продажи спортивных товаров. Он помогает пользователю быстро находить нужные товары и оформлять заказ, а администратору — управлять каталогом и заказами.

Это ответ на вопросы:

- актуальность;
- практическая значимость;
- кому полезен проект;
- какую проблему он решает.

## Блок 3. Цель и задачи

Цель проекта — разработать интернет-магазин спортивных товаров как информационную систему.

Основные задачи:

1. проанализировать предметную область;
2. сформировать требования;
3. спроектировать архитектуру;
4. разработать структуру базы данных;
5. описать API;
6. создать прототип интерфейса;
7. подготовить тестирование.

Это ответ на вопросы:

- какая цель;
- какие задачи;
- что вы сделали лично.

## Блок 4. Архитектура

В проекте используется клиент-серверная архитектура. Пользователь работает с frontend-частью, frontend обращается к backend через REST API, а backend взаимодействует с базой данных PostgreSQL.

Это ответ на вопросы:

- как устроена система;
- почему выбрана такая архитектура;
- что такое frontend, backend и API.

## Блок 5. База данных

В базе данных выделены основные сущности: пользователи, роли, категории, бренды, товары, корзины, заказы, позиции заказа, платежи и отзывы.

База данных реляционная, потому что между сущностями есть связи. Например, один заказ может содержать много товаров, а одна категория может включать много товаров.

Это ответ на вопросы:

- почему PostgreSQL;
- почему реляционная БД;
- какие таблицы есть;
- что такое нормализация;
- почему заказ и позиции заказа разделены.

## Блок 6. Что уже реализовано

В демонстрационном прототипе реализованы:

- главная страница;
- каталог товаров;
- поиск;
- фильтрация по категории и цене;
- добавление товаров в корзину;
- подсчет количества и общей суммы.

Это ответ на вопросы:

- что работает;
- что вы показываете на защите;
- что есть в практической части.

## Блок 7. Что можно улучшить

В дальнейшем проект можно расширить за счет полноценного backend, подключения реальной базы данных, авторизации пользователей, онлайн-оплаты и административной панели.

Это ответ на вопросы:

- что не реализовано;
- какие ограничения есть;
- как развивать проект дальше.

## Самая короткая версия, которую надо запомнить

Если совсем мало времени, выучите вот это:

Мой проект — это интернет-магазин спортивных товаров как информационная система. Его цель — автоматизировать онлайн-продажу товаров через каталог, поиск, фильтрацию, корзину и оформление заказа. В проекте выбрана клиент-серверная архитектура: frontend, backend, REST API и база данных PostgreSQL. Я спроектировал структуру системы, базу данных, API и реализовал демонстрационный интерфейсный прототип. В дальнейшем проект можно расширить за счет серверной части, авторизации и онлайн-оплаты.
