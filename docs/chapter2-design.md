# Глава 2. Проектирование системы

## 2.1 Общая архитектура

Для реализации интернет-магазина спортивных товаров выбрана клиент-серверная архитектура. Такой подход позволяет разделить систему на логические уровни и упростить дальнейшую разработку.

Клиентская часть отвечает за отображение интерфейса, взаимодействие с пользователем, ввод данных, поиск, фильтрацию и отображение результатов. Серверная часть должна принимать запросы от клиента, выполнять бизнес-логику, проверять данные, обращаться к базе данных и возвращать результат. База данных используется для хранения информации о пользователях, товарах, заказах и связанных сущностях.

Схему взаимодействия можно описать следующим образом:

1. Пользователь открывает веб-приложение.
2. Интерфейс отправляет запросы на сервер через REST API.
3. Сервер обрабатывает запрос и обращается к базе данных.
4. База данных возвращает необходимую информацию.
5. Сервер формирует ответ в формате JSON.
6. Клиент отображает результат пользователю.

Преимуществами выбранной архитектуры являются:

- четкое разделение ответственности между слоями;
- удобство масштабирования;
- возможность независимого развития интерфейса и серверной части;
- удобство тестирования.

## 2.2 Функциональные требования

Проектируемая система должна обеспечивать реализацию следующих функций:

1. Просмотр каталога спортивных товаров.
2. Отображение категорий и брендов.
3. Поиск товаров по названию.
4. Фильтрацию по категории и цене.
5. Просмотр карточки товара с описанием и ценой.
6. Добавление товара в корзину.
7. Изменение количества товаров в корзине.
8. Оформление заказа.
9. Хранение данных о заказах.
10. Управление товарами и заказами со стороны администратора.

## 2.3 Роли пользователей

В системе предусмотрены три основные роли:

### Гость

Гость может:

- просматривать каталог;
- искать товары;
- применять фильтры;
- знакомиться с карточкой товара.

### Зарегистрированный пользователь

Пользователь получает дополнительные возможности:

- добавление товара в корзину;
- оформление заказа;
- просмотр своей истории заказов.

### Администратор

Администратор отвечает за сопровождение магазина и может:

- создавать и редактировать товары;
- управлять категориями;
- контролировать остатки;
- изменять статусы заказов;
- просматривать общую информацию о продажах.

## 2.4 Проектирование базы данных

Для хранения данных используется реляционная модель. Она подходит для проекта, так как в системе присутствует большое количество связанных сущностей.

Основные таблицы:

- `users` — данные пользователей;
- `roles` — роли пользователей;
- `addresses` — адреса доставки;
- `categories` — категории товаров;
- `brands` — бренды;
- `products` — товары;
- `product_images` — изображения товаров;
- `carts` — корзины пользователей;
- `cart_items` — позиции корзины;
- `orders` — заказы;
- `order_items` — состав заказа;
- `payments` — данные об оплате;
- `reviews` — отзывы;
- `order_statuses` — статусы заказа.

Выбор такой структуры позволяет нормализовать данные и уменьшить дублирование информации. Например, категории и бренды вынесены в отдельные таблицы, поскольку один бренд или категория могут относиться ко многим товарам. Аналогично заказ и его состав хранятся раздельно, поскольку один заказ включает несколько позиций.

## 2.5 Проектирование API

Для взаимодействия клиентской и серверной частей выбран REST API. В этом подходе каждая группа сущностей представлена собственным набором адресов и HTTP-методов.

Основные направления API:

- аутентификация пользователей;
- получение категорий;
- получение списка товаров;
- получение карточки товара;
- работа с корзиной;
- оформление заказа;
- административные действия.

Пример запросов:

- `GET /api/products` — получить список товаров;
- `GET /api/products/{id}` — получить карточку товара;
- `POST /api/cart/items` — добавить товар в корзину;
- `POST /api/orders` — оформить заказ;
- `PATCH /api/admin/orders/{id}/status` — изменить статус заказа.

Использование API делает систему более гибкой, так как один и тот же сервер может обслуживать веб-версию, мобильное приложение или административный интерфейс.

## 2.6 Проектирование интерфейса

При проектировании интерфейса основное внимание уделялось удобству пользователя. Были выделены следующие ключевые страницы:

1. Главная страница.
2. Каталог товаров.
3. Карточка товара.
4. Корзина.
5. Форма оформления заказа.
6. Личный кабинет.
7. Административная панель.

Главная страница должна знакомить пользователя с магазином и быстро переводить его к каталогу. Каталог должен позволять легко находить товар за счет фильтров и поиска. Карточка товара должна давать достаточный объем информации для принятия решения о покупке. Корзина должна быть максимально простой и понятной.

## 2.7 Нефункциональные требования

Помимо основного функционала система должна удовлетворять следующим нефункциональным требованиям:

- адаптивность интерфейса;
- надежность хранения данных;
- корректная валидация форм;
- безопасность пользовательских данных;
- приемлемая производительность;
- возможность сопровождения и расширения проекта.

Выбрана клиент-серверная архитектура, определены роли пользователей, сформированы функциональные требования, продумана модель базы данных и описан API. Полученные решения создают основу для реализации и тестирования проекта.
